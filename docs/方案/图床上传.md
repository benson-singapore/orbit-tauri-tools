## 本项目后端实现概要

### 对外接口（Gin）

- 路由注册在 `back-end/routes/routes.go`：图床接口挂在 **`/api/v1/images`** 下，与 v1 其它接口一样经过 **Token 鉴权**（`authMiddleware.RequireToken()`）。
- **`POST /api/v1/images/upload`**（`UploadImage`）
    - 请求：`multipart/form-data`，字段名必须为 **`file`**。
    - 成功响应：`utils.Success`，业务数据形如 `data.image.url`（图床返回的预览/直链，与 Swagger 中 `models.ImageUploadResult` 一致）。
- **`POST /api/v1/images/upload-url`**（`UploadImageByURL`）
    - 请求：JSON，字段 **`url`**（可直链访问的图片地址），模型见 `models.UploadImageByURLRequest`。
    - 成功响应：同样返回 `data.image.url`。

### 控制器层（`back-end/controller/image_controller.go`）

- `ImageController` 内嵌 `*service.ImgbbService`（`NewImageController` 中创建），**不负责**与 ImgBB 直接对话，只处理 HTTP 入参、限流与校验后调用 Service。
- **单文件上限 20 MiB**（`maxImageUploadBytes`，注释说明与 ImgBB 官方上限相关）。
    - 对整个请求体使用 `http.MaxBytesReader`；`ParseMultipartForm` 使用该上限；读文件时用 `io.LimitReader` 防止实际读入超过 20MB。
    - `fh.Size` 与读入字节数双重校验，过大返回 400。
- **文件上传**额外用 `looksLikeImageExt` 校验扩展名（如 png、jpg、gif、webp、bmp、svg、ico、avif 等）；**无扩展名时仍放行**（交给下游尝试上传）。
- 解析失败、字段缺失、过大、非图片扩展名、或 Service 返回错误时，多返回 **400** 并带说明；打开/读取文件失败返回 **500**。

### 服务层与 ImgBB（`back-end/service/imgbb_service.go`）

- 实际上传由 **`ImgbbService`** 完成，与上文 curl 中 **`POST https://imgbb.com/json`** 的字段语义一致（文件走 `type=file` + `source` 文件体，URL 走 `type=url` + `source` 为 URL）。
- **策略（二选一）**
    1. 若设置环境变量 **`IMGBB_API_KEY`**：走 ImgBB **官方 API** `https://api.imgbb.com/1/upload`。
    2. 否则：走网页 **`/json` 接口**（与浏览器同源行为一致）。
- **网页接口路径下，默认无需手动配置**
    - 未设置 `IMGBB_API_KEY` 时，服务端会先 **`GET https://imgbb.com/`**，从 HTML 解析 `auth_token="..."`，并用 **Cookie Jar** 保存响应中的 **`PHPSESSID`**，再携带会话 `POST https://imgbb.com/json`。
- **可选手动会话**（自动获取失败、网络限制等）：设置 **`IMGBB_AUTH_TOKEN`** + **`IMGBB_PHPSESSID`**（或 **`IMGBB_COOKIE`**），与手动 curl 一致。

### 与抓包示例的对应关系

| 能力 | 上文 curl | 本项目入口 | Service 侧 |
|------|-----------|------------|------------|
| 本地文件 | `type=file`，`source` 为文件 | `POST /api/v1/images/upload`，字段 `file` | `UploadFileBytes` |
| 远程 URL | `type=url`，`source` 为 URL | `POST /api/v1/images/upload-url`，JSON `url` | `UploadFromURL` |
