这是一份专为**单人/小团队实战落地**重构的 `Universal Reader`（万能阅读器）实施方案。

本方案抛弃所有“PPT 架构”与非必要依赖，聚焦于**开发效率、跨平台编译便利性、核心技术闭环**。

---

## 一、 总体架构与通信模型

为了绝对避免 React 跨域问题，并保持 Rust 的轻量，采用 **“Frontend (UI) + Tauri (Shell) + Go (Core)”** 的三层架构。

### 1. 架构拓扑图

```
┌────────────────────────────────────────────────────────┐
│                   Tauri 2.0 Desktop Shell              │
│  ┌──────────────────────────┐  ┌────────────────────┐  │
│  │    React Frontend UI     │  │  Multi-Window Web  │  │
│  │   (Tailwind + shadcn)    │  │  (Cloudflare 挑战)  │  │
│  └────────────┬─────────────┘  └──────────┬─────────┘  │
└───────────────┼───────────────────────────┼────────────┘
                │ IPC (Tauri Plugin/Stdin)  │ Intercept Cookie
┌───────────────▼───────────────────────────▼────────────┐
│                    Go Runtime (Sidecar)                │
│  ┌──────────────────────────────────────────────────┐  │
│  │ 核心引擎: Goja(JS) / SQLite / Cache / Downloader │  │
│  └────────────────────────┬─────────────────────────┘  │
└───────────────────────────┼────────────────────────────┘
                            ▼
                     [ 互联网资源站 ]

```

### 2. 核心通信机制

* **控制流 (Control Flow)**：React $\rightarrow$ Tauri IPC $\rightarrow$ Go Runtime (Sidecar)。UI 的所有操作（如“点击搜索”）通过 Tauri 的 `sidecar` 管道或本地 RPC 转发给 Go。
* **数据流 (Data Flow)**：Go 处理完数据后返回 JSON 给前端渲染。
* **静态资源流 (Stream/Image Flow)**：Go 在本地随机端口启动一个 `localhost` HTTP Server，专门用于**图片缓存代理**和**视频切片流转发**。前端通过 `<img src="http://localhost:port/proxy?url=..." />` 加载资源，完美绕过 CORS。

## 二、 核心技术选型

### 前端 (Frontend)

* **框架**：React 19 + Vite + TypeScript（保证绝对的开发生态和 AI 代码生成友好度）。
* **UI 库**：TailwindCSS + shadcn/ui（支持高密度、紧凑的 Bento Grid 布局与暗黑模式）。

### 壳容器 (Desktop Shell)

* **框架**：Tauri 2.0
* **职责**：仅负责系统原生窗口管理、系统托盘（Tray）、文件本地权限、以及唤起 Go Sidecar。**不写任何 Rust 业务代码**。

### 后端核心 (Backend Runtime)

* **语言**：Go 1.24+
* **驱动/ORM**：`modernc.org/sqlite`（纯 Go 实现，**无 CGO 依赖**，跨平台交叉编译极其顺畅） + `bun` (轻量、高性能 SQL Builder)。
* **JS 引擎**：`github.com/dop251/goja`（纯 Go 实现的 ECMAScript 5.1+ 引擎，轻量且易于沙箱控制）。

## 三、 统一数据模型设计 (Data Models)

为了兼顾“聚合搜索”的统一性与“各媒体内容”的差异性，采用**基础模型 + 动态元数据**的设计。

### 1. 统一元数据模型 (Media)

```go
type Media struct {
    ID          string    `json:"id"`          // 插件ID + 资源内部ID
    PluginID    string    `json:"plugin_id"`   // 所属插件
    Title       string    `json:"title"`       // 标题
    Cover       string    `json:"cover"`       // 封面图 (Local Proxy URL)
    Type        string    `json:"type"`        // "manga", "video", "article", "novel"
    Description string    `json:"description"` // 简介
    UpdateAt    int64     `json:"update_at"`   // 更新时间
}

```

### 2. 详情多态扩展 (Detail)

不建议硬编码字段，通过 SQLite 的 `JSON` 类型存储差异化内容：

```go
type MediaDetail struct {
    Media
    Author      string          `json:"author"`
    Status      string          `json:"status"` // 连载/完结
    Catalog     bun.JSONString  `json:"catalog"` // 目录：Manga为章节列表，Video为剧集/清晰度，Article为空
}

```

## 四、 核心子系统架构

### 1. 插件系统 (Plugin Host)

插件必须完全运行在 Go 层的 Goja 沙箱中。

* **隔离原则**：插件禁止直接调用网络和文件系统。
* **能力注入**：Go 向 Goja 环境注入 `host` 对象：

```javascript
// 插件编写范式 (index.js)
async function search(keyword) {
    const rawHtml = await host.request({
        url: `https://example.manga.com/search?q=${encodeURIComponent(keyword)}`,
        method: "GET",
        headers: { "User-Agent": host.userAgent }
    });
    return host.parseHtml(rawHtml, {
        title: "div.title -> text",
        cover: "div.cover img -> attr(src)"
    });
}

```

### 2. Cloudflare 穿透方案 (CF Engine)

* **触发机制**：Go Runtime 发起请求 $\rightarrow$ 收到 `503` 或 `403` (包含 `cf-cloudflare` 标记) $\rightarrow$ 返回 `ERR_NEED_CF_CHALLENGE` 错误给前端。
* **通关流程**：
1. React 收到错误，调用 Tauri 创建一个**独立的原生 WebView 窗口**，直接加载目标资源站。
2. 该窗口允许用户手动完成人机验证（Turnstile/Challenge）。
3. 验证通过后，Go 层的 `CookieJar` 拦截并同步该窗口的 `cf_clearance` Cookie 和 `User-Agent`。
4. 关闭独立窗口，Go 自动重试先前失败的请求，React 界面无缝刷新。



### 3. 缓存与下载管理 (Cache & Download)

* **图片缓存**：Go 接收到前端请求后，下载图片至 `/cache/images/`，进行哈希命名。前端通过本地代理读取，支持离线浏览。
* **视频流媒体**：集成本地 `mpv` 动态库或通过 Tauri 窗口挂载。对于 `m3u8`，Go 负责多线程下载分片并本地解密，合并为本地伪流媒体服务供前端播放。

## 五、 模块化目录结构

单人开发必须保证目录结构“看一眼就知道代码在哪”。

```text
universal-reader/
├── app/                  # Tauri 壳工程
│   ├── src-tauri/        # Rust 核心（仅放 sidecar 配置、窗口管理、打包脚本）
│   └── src/              # React 前端 UI (Vite 驱动)
│       ├── src/components/  # shadcn 紧凑型组件
│       └── src/pages/       # MangaReader, VideoPlayer, RSSFeed, Setting
├── runtime/              # Go 核心后端 (编译为 Sidecar 执行文件)
│   ├── cmd/main.go       # 入口，处理 IPC 管道/本地 RPC 与本地 HTTP 代理
│   ├── internal/
│   │   ├── api/          # 响应前端控制指令的 Handler
│   │   ├── plugin/       # Goja 引擎、插件生命周期、上下文注入
│   │   ├── db/           # SQLite 连接、Bun ORM 初始化、Migation
│   │   ├── cf/           # CookieJar 管理与 Cloudflare 拦截器
│   │   ├── cache/        # 本地文件缓存逻辑 (Images/Chunks)
│   │   └── downloader/   # 多线程下载任务队列 (Manga图片/m3u8)
└── plugins/              # 插件市场与本地调试目录
    └── base-manga-plugin/
        ├── manifest.json # 插件元数据（名称、版本、支持类型）
        └── index.js      # 纯 JS 抓取解析脚本

```

## 六、 敏捷落地四阶段计划

### Phase 0：技术闭环验证（第 1 周，核心目标：打通三层架构）

* [ ] 初始化 Tauri 2.0 + React 19 工程。
* [ ] 编写 Go 最小 Core，实现通过 Stdin/Stdout 接收 Tauri 指令并返回 `"hello world"`。
* [ ] Go 成功内嵌 Goja，运行一段简单的 JS 字符串并输出结果。
* **通过标准**：点击前端按钮，能触发 Go 运行 JS，并在前端看到输出。

### Phase 1：漫画与 RSS 核心（第 2~4 周，核心目标：做出可用的 MVP）

* [ ] 实现 Go 层的 SQLite 本地化存储（历史、收藏）。
* [ ] 完善 `host.request` 注入，编写第一个真正的漫画/RSS 插件。
* [ ] 实现本地图片代理与前端长图、分页阅读器。
* **不做**：视频、高级防爬、同步、插件市场。

### Phase 2：CF 破盾与聚合搜索（第 5~7 周，核心目标：解决稳定抗爬问题）

* [ ] 实现 Tauri 多窗口联动，完成手动 Cloudflare 验证与 Cookie 提取。
* [ ] 实现 Go 层并发控制，支持在搜索框输入关键词时，多插件同时异步抓取并聚合返回。
* [ ] 支持本地 `.zip` 插件包的手动导入。

### Phase 3：视频与流媒体支持（第 8~11 周，核心目标：攻克最高复杂度）

* [ ] 引入 `mpv` 挂载或前端成熟播放器，对接 Go 本地 m3u8 切片流代理。
* [ ] 编写视频资源站插件，支持选集、多线路解析。
* [ ] 引入多线程下载队列，支持视频/漫画的批量离线下载。

### Phase 4：生态化与 AI（第 12 周+）

* [ ] 插件市场 Registry 自动化。
* [ ] 接入大模型 API，实现 RSS/新闻/小说的本地 AI 摘要与一键翻译。

## 七、 个人开发死守原则

1. **绝对不要在 React 里直接 Fetch 资源站**。
2. **拒绝 CGO**，所有 Go 依赖必须能一键 `GOOS=windows GOARCH=amd64 go build`。
3. **元数据以极简为主**，详情页全部走 JSON 动态解析，不要为了几颗杂碎字段频繁修改 SQLite 表结构。
4. **先做核心链路，再做体验优化**。Phase 1 结束前，界面用最朴素的单色暗黑主题即可，不要在动效上浪费时间。

这个方案移除了所有技术虚荣心，每一行都为了**单人高效率交付**而设计，完全具备工业落地可行性。你想好第一个要接入的资源站了吗？
